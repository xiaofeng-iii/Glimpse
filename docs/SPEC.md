# Glimpse 项目测试标准规范

## 1. 文档概述

### 1.1 目的

本文档为 Glimpse 项目提供统一的测试标准，采用 pytest 作为测试框架，通过「单元测试 + 集成测试」的分层策略，确保代码质量和项目可维护性。

### 1.2 适用范围

- 全体项目成员（后端工程师、前端工程师、QA 工程师）
- 所有提交到仓库的代码必须通过相应测试

---

## 2. 测试目录结构

### 2.1 整体架构

**注：单元测试内容由具体的模块实现，这里仅定义目录结构。具体实现请参考 `tests/unit/` 目录下的文件**

```
tests/
├── unit/                      # 单元测试：测试最小可测试单元
│   ├── conftest.py            # 单元测试 fixtures 配置
│   ├── config/                # 配置模块测试
│   ├── core/                  # 核心模块测试
│   ├── db/                    # 数据库模块测试
│   ├── services/              # 服务模块测试
│   └── ui/                    # UI 模块测试
│
├── integration/               # 集成测试：测试模块间交互
│   ├── conftest.py            # 集成测试 fixtures 配置
│   ├── test_capture_to_storage.py      # 截图→存储流程
│   ├── test_ai_analysis_pipeline.py   # AI 分析流程
│   ├── test_search_pipeline.py         # 搜索功能流程
│   ├── test_database_sync.py           # 双数据库同步
│   └── test_end_to_end.py              # 端到端测试
│
├── fixtures/                  # 测试数据和 fixtures 共享资源
│   ├── sample_images/         # 测试用截图样本
│   └── sample_responses/     # AI API 模拟响应数据
│
├── helpers/                  # 测试辅助工具
│   ├── mock_services.py       # 服务 Mock 类
│   └── test_data_generator.py # 测试数据生成器
│
├── conftest.py                # pytest 全局配置
└── pytest.ini                # pytest 配置文件
```

### 2.2 目录对应关系

| 测试目录 | 源码目录 | 说明 |
|---------|---------|------|
| `tests/unit/config/` | `config/` | 配置模块的单元测试 |
| `tests/unit/core/` | `core/` | 核心模块的单元测试 |
| `tests/unit/db/` | `db/` | 数据库模块的单元测试 |
| `tests/unit/services/` | `services/` | 服务模块的单元测试 |
| `tests/integration/` | — | 跨模块集成测试 |

### 2.3 命名约定

| 类型 | 命名模式 | 示例 |
|-----|---------|------|
| 测试文件 | `test_<模块名>.py` | `test_capture.py` |
| 测试类 | `Test<被测模块>` | `TestAIClient` |
| 测试函数 | `test_<功能>_<场景>_<预期>` | `test_save_memory_success` |
| Fixture | `<描述性名称>` | `sample_memory_record` |

## 3. 测试分类

### 3.1 单元测试 (Unit Testing)

#### 3.1.1 定义

单元测试是对软件系统中的最小可测试单元进行验证的测试活动。在面向对象编程中，最小单元通常指一个函数或一个类的方法。

#### 3.1.2 特点

| 特性 | 说明 |
|-----|------|
| 隔离性 | 每个测试独立运行，不依赖其他测试 |
| 快速性 | 单个测试通常在毫秒级完成 |
| 可重复 | 相同输入产生相同输出 |
| 明确性 | 每个测试验证一个具体行为 |

#### 3.1.3 适用场景

- 验证独立函数的功能正确性
- 验证类的属性和方法行为
- 验证配置解析和路径处理逻辑
- 验证数据转换和格式化

#### 3.1.4 示例场景

```
场景：测试 SQLite 管理器的 save_memory 方法
输入：有效的记忆数据（摘要、来源、标签）
预期：方法返回记忆 ID，数据成功写入数据库
```

### 3.2 集成测试 (Integration Testing)

#### 3.2.1 定义

集成测试是对多个单元进行组合验证的测试活动，用于测试模块间的交互和数据流动是否符合预期。

#### 3.2.2 特点

| 特性 | 说明 |
|-----|------|
| 真实性 | 尽可能使用真实组件，减少 Mock |
| 复杂性 | 涉及多个模块和外部依赖 |
| 数据隔离 | 每个测试使用独立数据，执行后清理 |
| 流程性 | 通常对应完整的业务流程 |

#### 3.2.3 适用场景

- 验证数据库双写一致性（SQLite + ChromaDB）
- 验证截图捕获到存储的完整流程
- 验证 AI 分析管道的端到端行为
- 验证语义搜索的准确性和排序

#### 3.2.4 示例场景

```
场景：测试「截图 → AI 分析 → 双数据库存储」完整流程
输入：一张测试截图
预期：
  1. 截图被正确压缩和处理
  2. AI 返回有效的摘要内容
  3. SQLite 存储了完整元数据
  4. ChromaDB 存储了向量数据
  5. 两库通过 UUID 正确关联
```

### 3.3 测试策略对比

| 维度 | 单元测试 | 集成测试 |
|-----|---------|---------|
| 测试范围 | 单个函数/方法 | 多模块交互 |
| 执行速度 | 快（毫秒级） | 较慢（秒级） |
| Mock 使用 | 高（隔离外部依赖） | 低（使用真实组件） |
| 覆盖率要求 | ≥85% | ≥70% |
| 维护成本 | 较低 | 较高 |

## 4. pytest 配置

### 4.1 pytest.ini 配置项

```ini
[pytest]
testpaths = tests                              # 测试发现路径
python_files = test_*.py                      # 测试文件匹配
python_classes = Test*                        # 测试类匹配
python_functions = test_*                     # 测试函数匹配

addopts =                                      # 运行选项
    -v                                         # 详细输出
    --tb=short                                 # 简短回溯
    --cov=.                                    # 覆盖率统计
    --cov-report=html                          # HTML 报告

markers =                                      # 测试标记
    unit: 单元测试
    integration: 集成测试
    slow: 运行较慢的测试
```

### 4.2 Fixture 作用域

| 作用域 | 生命周期 | 适用场景 |
|-------|---------|---------|
| `session` | 整个测试会话 | 全局共享资源（项目路径、数据库连接） |
| `module` | 整个测试模块 | 模块内共享数据 |
| `class` | 测试类内 | 类的所有测试方法 |
| `function` | 单个测试函数 | 每次测试重新创建 |

## 5. 测试设计原则

### 5.1 单元测试设计原则

| 原则 | 说明 |
|-----|------|
| 单一职责 | 每个测试函数只验证一个行为 |
| AAA 模式 | Arrange（准备）→ Act（执行）→ Assert（断言） |
| 独立性 | 测试间无依赖，可任意顺序执行 |
| 可读性 | 测试名称清晰表达意图 |

### 5.2 集成测试设计原则

| 原则 | 说明 |
|-----|------|
| 真实环境 | 优先使用真实组件，减少 Mock |
| 数据隔离 | 测试数据独立，执行后清理 |
| 边界清晰 | 明确测试的起点和终点 |
| 性能关注 | 设置合理超时，监控执行时间 |

### 5.3 测试命名规范

```
test_<被测模块>_<操作>_<条件>_<预期结果>

示例：
test_ai_client_send_request_success
test_sqlite_save_memory_duplicate_handled
test_search_query_returns_sorted_results
```

## 6. Mock 与 Stub 使用

### 6.1 使用场景

| 场景 | 单元测试 | 集成测试 |
|-----|---------|---------|
| 外部 API | ✅ 必须 Mock | ❌ 不 Mock（测试真实交互） |
| 数据库 | ✅ Mock（SQLite） | ❌ 不 Mock（测试真实 DB） |
| 文件系统 | ✅ Mock（tmp_path） | ⚠️ 视情况 |
| 时间相关 | ✅ 必须 Mock | ⚠️ 视情况 |

### 6.2 Mock 对象类型

| 类型 | 说明 | 示例 |
|-----|------|------|
| Mock | 通用模拟对象 | `Mock(return_value=...)` |
| AsyncMock | 异步函数模拟 | `AsyncMock(return_value=...)` |
| PropertyMock | 属性模拟 | `PropertyMock(return_value=...)` |
| patch | 上下文管理器 | `@patch("module.Class")` |

## 7. 测试覆盖率要求

### 7.1 覆盖率目标

| 测试类型 | 分支覆盖率 | 行覆盖率 |
|---------|-----------|---------|
| 单元测试 | ≥85% | ≥90% |
| 集成测试 | ≥70% | ≥80% |
| 整体 | ≥75% | ≥85% |

### 7.2 覆盖率报告

测试运行后自动生成报告：

```
tests/reports/
├── coverage/index.html    # HTML 覆盖率报告
├── coverage.xml           # CI/CD 集成报告
└── .coverage              # 覆盖率数据文件
```

## 9. 最佳实践

### 9.1 测试代码质量

| 实践 | 说明 |
|-----|------|
| 避免测试逻辑 | 测试代码应简单直接 |
| 描述性断言 | 使用清晰的断言消息 |
| 边界值测试 | 测试正常、边界、异常情况 |
| 参数化测试 | 使用 `@pytest.mark.parametrize` 减少重复 |

### 9.2 异常测试

| 场景 | 验证点 |
|-----|-------|
| 无效输入 | 抛出预期异常类型 |
| 边界条件 | 处理临界值正确 |
| 错误恢复 | 部分失败时数据一致性 |

### 9.3 性能测试

| 指标 | 要求 |
|-----|------|
| 单元测试执行时间 | ≤1 秒/个 |
| 集成测试执行时间 | ≤10 秒/个 |
| 搜索响应时间 | ≤500ms（大规模数据集） |
